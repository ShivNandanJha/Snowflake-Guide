Foundations

ETL vs ELT

How traditional ETL worked

Traditional ETL followed a strict sequence: Extract → Transform → Load. Data was cleaned and reshaped before it ever entered the warehouse. This made sense when storage was expensive and compute was limited — you only stored what you absolutely needed in its final form.

The old pipeline

Source → [Transform on separate server] → Warehouse (clean data only)

Why ETL evolved into ELT

Cloud data warehouses like Snowflake, BigQuery, and Redshift changed the economics. Storage became cheap and compute became massively scalable. So the transformation step moved inside the warehouse itself.

Old (ETL)
Transform before load
New (ELT)
Load raw, transform inside warehouse
Who transforms?
Separate ETL server
Who transforms?
The warehouse itself (dbt)
Key insight

ELT pipelines are simpler to build, easier to debug, and far more flexible — because you always have the original data to re-transform from.

Why raw data is valuable

Raw data serves as the permanent source of truth. When business definitions change, transformations fail, or new requirements emerge — you can rebuild all downstream models from the original data without needing to re-extract from source systems.

Biggest advantage

If a transformation has a bug, or Finance changes the revenue definition, you can fix the dbt model and re-run. No need to go back to the source. The raw data is already there.

What happens without raw data?

If you only stored transformed data and a business rule changes, you'd need to re-extract from the source — which might be slow, rate-limited, or no longer available in the same form.

dbt

Why business logic shouldn't live in Power BI

When revenue calculations are duplicated across three dashboards, you get three different answers the moment Finance changes the definition. That's chaos.

The problem scenario

Sales Dashboard, Finance Dashboard, and Marketing Dashboard each compute revenue independently. Finance updates the formula → 3 dashboards need updating → risk of inconsistency → analysts lose trust in data.

Traditional analyst thinking
"How do I build this dashboard?"
Analytics engineer thinking
"How do I create a trusted dataset that powers ALL dashboards?"
The solution

Business logic lives in dbt as a single source of truth. All dashboards pull from the same fct_orders or dim_customers model. Update once → everywhere is consistent.

dbt is also more than just SQL files. It provides testing, documentation, lineage tracking, and version control — making data transformations as robust as software engineering.

Grain vs metric

Core concept

Grain ≠ Metric. Grain answers: "What does one row represent?" A metric is a calculation on top of that.

order_idcustomer_idamount
1101500
2102700

Each row = one order. Therefore: Grain = One Order.

Snowflake

Snowflake architecture

Snowflake uses a multi-cluster shared-data architecture with three distinct layers that can scale independently.

Cloud services (the brain)
Query parsing, optimization, metadata, access control — runs continuously
↓
Virtual warehouses (compute)
Independent clusters per team (Finance, Marketing, Analytics) — runs only when executing queries
↓
Storage layer
Centralized, compressed, columnar storage — always on (Orders, Customers, Products, Payments)

Why separate storage and compute?

An e-commerce company might store 10TB of data 24/7, but employees only run reports from 9 to 6. In a traditional database, storage and compute are tied together — you pay for both even at 2 AM when nobody is querying.

Snowflake's cost advantage

With Snowflake, you get storage + on-demand compute. Store data all month, but pay for compute only when queries actually run — a huge cost saving.

One-minute interview answer

Snowflake uses a multi-cluster shared-data architecture with three layers: Storage, Compute, and Cloud Services. Data is stored centrally while compute is provided by independent virtual warehouses. This allows organizations to scale storage and compute independently, avoid resource contention between teams, and pay only when compute is actually needed.

Stages & data loading

A Stage is a temporary holding location for data files before they are loaded into Snowflake tables.

Internal stage
Managed by Snowflake itself
External stage
S3, Azure Blob Storage, or GCS
Loading command

Data is loaded from a stage into tables using the COPY INTO command.

Stages decouple ingestion from the warehouse — files can be staged and validated before being permanently loaded.

Time travel & Fail-safe

Time travel

Time Travel lets you access historical versions of data and recover dropped or modified objects within a configured retention period. You can query past data using AT or BEFORE clauses, and recover objects using UNDROP.

Fail-safe

Fail-safe is a Snowflake-managed disaster recovery mechanism. Unlike Time Travel, users cannot directly access or query Fail-safe data — it exists purely as a backend safety net managed by Snowflake.

Snowflake

Zero-copy cloning

Zero-Copy Cloning lets you clone databases, schemas, and tables instantly — without physically copying the underlying data. Storage is only consumed when changes are made to the cloned object.

How it works

The clone shares the same underlying micro-partitions as the source. Only when you write new data or update existing data does additional storage get consumed — making cloning essentially free until you diverge.

When to use it

Development environments Testing new dbt models Safe experimentation Backup before migrations

Semi-structured data

VARIANT

VARIANT is Snowflake's semi-structured data type. It can store JSON, arrays, and objects — allowing Snowflake to work directly with semi-structured data without requiring a fixed schema upfront.

Example use case

App event logs often arrive as JSON blobs. Store them in a VARIANT column, then use dbt to extract and normalize only the fields you need into structured tables.

FLATTEN

FLATTEN converts JSON arrays into relational rows, making nested data easier to query and analyze. Each array element becomes its own row.

QUALIFY

QUALIFY filters the results of window functions — similar to how HAVING filters aggregates. It lets you write window function conditions without a subquery.

Should you expose VARIANT columns directly to Power BI?

No. Analytics Engineers first extract and normalize the required fields into structured tables that are easier to query, test, document, and model. Power BI always consumes structured, dbt-modelled data.

Performance

Micro-partitions & clustering

Micro-partitions

A micro-partition is Snowflake's internal storage unit. Snowflake automatically organizes data into micro-partitions and stores metadata (minimum and maximum values, row counts, statistics) about each partition.

Each micro-partition stores
Rows + Metadata + Statistics
Size range
50–500 MB compressed

Partition pruning

Partition pruning is the process of skipping micro-partitions that cannot satisfy a query predicate. Instead of scanning the whole table, Snowflake reads only relevant partitions — dramatically reducing data scanned and improving performance.

Clustering keys

For very large tables that are frequently filtered on specific columns (e.g., ORDER_DATE or REGION), you can define clustering keys. These help Snowflake maintain better micro-partition organization, improving partition pruning effectiveness.

When to use clustering keys

Only for very large tables (>1TB) that are frequently filtered on specific high-cardinality columns. Clustering keys have a maintenance cost — don't use them by default.
